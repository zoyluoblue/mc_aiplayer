# AGENT.md

本文件定义本仓库后续开发代理需要遵守的项目约定。目标是持续把 AI 玩家做成一个可信的 Minecraft 生存玩家，而不是一个能凭空修改世界的工具。

## 项目定位

这是一个面向 Minecraft Java Edition 的 Fabric AI 玩家模组。

目标版本固定为：

- Minecraft: `1.21.3`
- Fabric Loader: `0.18.4`
- Fabric API: `0.114.1+1.21.3`
- Java: `21`

除非用户明确要求，不要升级 Minecraft、Fabric Loader、Fabric API、Fabric Loom、Yarn mappings 或 Java 版本。

## 当前目标

AI 玩家应当像普通生存玩家一样行动：

- 初始没有免费材料。
- 拥有自己的背包。
- 需要移动到目标附近才能交互。
- 需要砍树、挖矿、合成、取物、放置、使用工作台和熔炉。
- 建造和制作必须消耗真实背包或箱子中的材料。
- 不允许飞行、瞬移、无敌、创造模式放置、凭空生成物品。

DeepSeek 负责理解意图、生成高层计划和复盘；模组代码负责查询游戏事实、验证计划并执行普通生存动作。

## 开发边界

优先沿用现有结构：

- `src/main/java/com/aiplayer/action/`：任务执行和动作调度。
- `src/main/java/com/aiplayer/action/actions/`：具体动作，例如采集、合成、建造、跟随。
- `src/main/java/com/aiplayer/entity/`：AI 玩家实体、背包和实体管理。
- `src/main/java/com/aiplayer/llm/`：DeepSeek 接口、提示词、响应解析、任务规划。
- `src/main/java/com/aiplayer/util/`：生存交互工具方法。
- `src/main/java/com/aiplayer/client/`：AI 面板、快捷键、客户端渲染。
- `docs/`：设计文档和阶段规划。

不要重建整个项目。除非确实需要，不要引入新框架或大规模重构。

## 命令约定

游戏命令使用英文子命令，说明文字可以是中文。

示例：

```mcfunction
/ai spawn
```

召唤当前玩家自己的 AI 玩家。

```mcfunction
/ai say 做一个门
```

向 AI 玩家发送自然语言任务。

```mcfunction
/ai stop
```

停止当前任务，并让 AI 玩家按正常导航逻辑回到玩家身边。

不要再把 `/ai spawn`、`/ai say`、`/ai stop` 等子命令改成中文。

## AI 面板约定

`Alt+2` 打开 AI 面板，macOS 为 `Option+2`。

AI 面板应保留这些基础能力：

- 输入自然语言任务。
- 显示 AI 消息和系统消息。
- 显示背包按钮，最多展示 20 个格子。
- 显示停止按钮，触发 `/ai stop`。

新增面板功能时，保持中文界面文案，但不要改变游戏命令语义。

## DeepSeek 使用边界

DeepSeek 可以做：

- 解析玩家意图。
- 在多个可行目标之间做选择。
- 基于观察数据生成高层计划。
- 在执行失败后提出下一步修正方向。

DeepSeek 不应该做：

- 编造 Minecraft 配方。
- 编造物品 ID。
- 决定材料数量的事实。
- 假设背包或箱子中存在物品。
- 直接输出会绕过生存规则的动作。

配方、材料数量、工具要求、方块存在性、箱子内容和背包内容必须由游戏代码查询和验证。

## 规划架构方向

后续优先按以下设计推进：

1. Observe 观察层：导出 AI 背包、附近箱子、附近方块、附近实体、当前位置和当前任务。
2. RecipeResolver 配方递归层：用 Minecraft 真实配方递归计算材料链。
3. DeepSeek Planner 规划层：基于观察结果和配方链生成高层计划。
4. Execute And Replan 执行复盘层：执行一个小步骤后重新观察，必要时让 DeepSeek 复盘。

详细路线见：

```text
docs/phases.md
```

实现时不要直接跳到“让 DeepSeek 完全接管执行”。先把观察、验证、执行做稳，再让 DeepSeek 参与决策。

## 生存执行规则

所有世界交互必须符合普通生存玩家语义：

- 破坏方块前必须移动到可触及距离。
- 放置方块前必须消耗背包或箱子中的对应物品。
- 合成前必须拥有材料。
- 挖石头或矿物前必须拥有合适工具。
- 需要工作台或熔炉的配方，必须先找到或制作对应设施。
- 如果目标不可达，要标记失败并换目标或请求重新规划。

世界读取和修改必须在 Minecraft server thread 执行。DeepSeek 和 HTTP 请求必须异步，不能阻塞 server thread。

## 背包和箱子规则

AI 自身背包是主要物品来源。

附近箱子可以作为额外物品来源，但必须满足：

- 箱子在扫描半径内。
- 路径可达。
- 操作前记录取出或放入的物品。
- 不允许假设箱子拥有未扫描到的物品。

后续实现箱子交互时，应提供明确动作：

- `withdraw_chest`
- `deposit_chest`
- `inspect_chest`

## 文档同步

改动用户可见行为时必须同步文档。

优先更新：

- `README.md`：命令、配置、玩法说明。
- `docs/`：设计路线、阶段计划、架构说明。

命令示例使用英文命令写法，后面用中文解释用途和含义。

## 配置规则

配置文件统一使用：

```text
config/ai.json
```

开发运行 `runClient` 时，配置会复制到：

```text
run/config/ai.json
```

不要引入新的 API key 文件名。不要把真实 API key 写入仓库。

## 验证命令

代码改动后默认运行：

```bash
./gradlew build
```

如果改动只涉及 Markdown 文档，可以不跑构建，但最终回复中要明确说明没有运行构建。

如果构建失败：

1. 阅读完整错误。
2. 优先修复当前改动导致的问题。
3. 最多连续修复 3 轮。
4. 仍失败时停止继续开发，并说明失败原因、相关文件和下一步建议。

## Phase 验证流程

按 `docs/phases.md` 开发时，必须逐个 phase 推进。

- 每完成一个 phase，先运行该 phase 对应的本地验证，例如单元测试、`./gradlew build` 或明确的实机复测步骤。
- 本地验证通过后，必须让 `agent2` 做只读验证，检查逻辑可行性、边界条件、回归风险和文档状态。
- `agent2` 只能审查和提出问题，不直接改代码。
- 如果 `agent2` 提出必须修的问题，先回到当前 phase 修复，再重新运行验证，并再次交给 `agent2` 复查。
- 当前 phase 的本地验证和 `agent2` 验证都通过后，才能在 `docs/phases.md` 中把该 phase 标记为 `[x] 已完成`，并继续下一个 phase。
- 最终回复中要说明完成了哪些 phase、运行了哪些验证，以及 `agent2` 是否还有剩余必须修项。

## 代码风格

保持 Java 代码简单直接：

- 优先小类和明确方法名。
- 不要为了抽象而抽象。
- 不要把 DeepSeek 返回 JSON 的字段散落在多个地方，优先集中 schema 和解析逻辑。
- 不要在动作类里硬编码大量配方，配方应逐步迁移到 `RecipeResolver`。
- 不要在客户端代码中执行服务端世界修改。

注释只写有助于理解复杂逻辑的内容，避免解释显而易见的代码。

## Git 和文件安全

工作区可能已有用户改动。

- 不要执行 `git reset --hard`。
- 不要执行会清除用户文件的命令。
- 不要回退不属于当前任务的文件。
- 看到不相关改动时忽略，不要顺手整理。
